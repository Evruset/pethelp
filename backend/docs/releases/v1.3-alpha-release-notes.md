# Release Notes: VetHelp v1.3-alpha

**Дата релиза:** 24 июня 2026 г.  
**Статус:** Release Candidate для контролируемого закрытого Alpha-тестирования.  
**Ветка-кандидат:** `feat/marketplace-alpha` → `main` после одобрения и merge PR #18.  
**Подтверждение CI:** Backend B-01 run #60: build, migration verification, platform E2E smoke, integration suites и B-01 k6 прошли успешно.

> Этот документ фиксирует готовность программного релиз-кандидата. Выход в Alpha зависит также от операционных gate'ов: доступа к Secret Manager, доступности PostgreSQL/LiveKit/эквайринга, сетевых политик и одобрения PR #18.

## 1. Сводка изменений

VetHelp v1.3-alpha формирует транзакционное ядро маркетплейса, бронирования, платежей и телемедицины.

- **Booking Core:** блокировки строк, ограничение `lock_timeout = 50ms`, контроль вместимости и durable outbox.
- **Security & AuthZ:** owner и клинический контекст вычисляются из JWT principal; доступ сотрудника дополнительно сверяется с активным membership локации в БД.
- **Marketplace Alpha:** 15-минутный SLA ручного подтверждения клиники уровня C, автоматическое освобождение нарушенных holds и безопасная ротация альтернативного слота.
- **Payments:** accepted alternative остаётся в `MIS_HELD`; исходный слот освобождается, а замещающий превращается в `BOOKED` только после подтверждённой авторизации платежа.
- **Telemedicine:** событие `CONFIRMED → telemed.session.start.requested.v1` передаётся через outbox; worker создаёт session в `WAITING_FOR_DOCTOR`.
- **Observability:** исходный `correlationId` переносится через HTTP, SQL/outbox, audit и telemed session. Критические operational alerts отправляются асинхронным Alert Forwarder.
- **Проверка:** `backend/test/platform-smoke.e2e-spec.ts` выполняет последовательность OWNER → Level-C Clinic → signed payment webhook → Telemed Worker на реальном PostgreSQL.

## 2. Предварительные условия

Перед раскаткой должны быть выполнены следующие условия.

1. PR #18 рассмотрен и смержен в `main`, либо Alpha разворачивается с одобренного commit SHA ветки `feat/marketplace-alpha`.
2. В кластере существует namespace `vethelp-alpha`.
3. Установлен External Secrets Operator и настроен `ClusterSecretStore/vethelp-platform-secrets`.
4. Registry image доступен кластеру. Alpha deployment сейчас использует `ghcr.io/evruset/pethelp-backend:alpha`; перед более широким контуром замените mutable tag на immutable image digest.
5. PostgreSQL 15/16 доступен по TLS/внутренней сети из namespace приложения.
6. LiveKit и acquiring endpoints доступны только по разрешённым egress policy.

## 3. Подготовка секретов

Примените ExternalSecret/ConfigMap из репозитория:

```bash
kubectl apply -f backend/k8s/alpha/secrets.yaml
kubectl -n vethelp-alpha wait \
  --for=condition=Ready externalsecret/vethelp-backend-secret \
  --timeout=120s
kubectl -n vethelp-alpha get secret vethelp-backend-secret
```

`vethelp-backend-secret` должен быть материализован из одобренного Secret Manager и содержать как минимум:

- `DATABASE_URL`
- `JWT_SECRET`
- `WORKER_SERVICE_TOKEN`
- `MIS_VET_MANAGER_API_KEY`
- `ACQUIRING_API_KEY`
- `ACQUIRING_WEBHOOK_SECRET`
- `LIVEKIT_API_KEY`
- `LIVEKIT_API_SECRET`
- `TELEGRAM_BOT_TOKEN` и `TELEGRAM_CHAT_ID` для Telegram Alert Forwarder **или** `SLACK_WEBHOOK_URL` для Slack

Значения `JWT_ISSUER`, `JWT_AUDIENCE`, `MIS_VET_MANAGER_BASE_URL`, `ACQUIRING_API_BASE_URL`, `LIVEKIT_API_URL`, флаги workers и настройки alert forwarder задаются через ConfigMap в том же manifest.

## 4. Накат миграций

Миграции должны быть применены **до** запуска новых pod'ов. Выполняйте их из доверенного runner/bastion с доступом к той же Alpha БД.

```bash
cd backend
npm install
npm run migrate:up
npm run migrate:verify
```

Критерии успеха:

- существует `public.schema_migrations`;
- checksum ledger `public.schema_migration_checksums` проходит `npm run migrate:verify`;
- присутствуют схемы `identity_schema`, `pet_schema`, `clinic_schema`, `booking_schema`, `payment_schema`, `telemed_schema` и `audit_schema`;
- миграции Marketplace Alpha и correlation propagation применены без ошибок.

## 5. Раскатка backend

Текущий Docker build definition расположен в `backend/Dockerfile`: это multi-stage image на `node:22-alpine`, runtime работает под непривилегированным `USER node`.

```bash
# Выполняется в CI/CD, пример локальной сборки:
docker build -f backend/Dockerfile -t ghcr.io/evruset/pethelp-backend:alpha backend

docker push ghcr.io/evruset/pethelp-backend:alpha

kubectl apply -f backend/k8s/alpha/deployment.yaml
kubectl -n vethelp-alpha rollout status deployment/vethelp-backend --timeout=180s
kubectl -n vethelp-alpha get pods -l app.kubernetes.io/name=vethelp-backend
```

Manifest создаёт Deployment, Service и ServiceAccount. Pod запускается с `runAsNonRoot`, `readOnlyRootFilesystem`, `allowPrivilegeEscalation: false`, dropped Linux capabilities и ограничениями ресурсов.

Проверка готовности:

```bash
kubectl -n vethelp-alpha port-forward service/vethelp-backend 3000:80
curl --fail --silent http://127.0.0.1:3000/v1/health
```

`/v1/health` выполняет проверку доступности БД; deployment использует этот endpoint для startup, readiness и liveness probes.

## 6. Сетевая безопасность: обязательный эксплуатационный gate

В текущем PR отсутствуют готовые репозиторные manifests для Ingress, webhook ingress и NetworkPolicy. Поэтому до внешней публикации Alpha Operations должна применить отдельные, прошедшие review manifests со следующими условиями:

1. публично доступны только требуемые API и health endpoints;
2. webhook endpoint эквайринга ограничен актуальными CIDR платёжного провайдера и проверяется HMAC в приложении;
3. административные и internal endpoints не маршрутизируются извне;
4. egress backend ограничен PostgreSQL, MIS, acquiring, LiveKit, Secret Manager и alert channel;
5. ingress/TLS/WAF и rate limits соответствуют правилам Alpha-контура.

Не считайте внешний контур готовым, пока эти объекты не применены и не подтверждены командой эксплуатации.

## 7. Постдеплой smoke-проверка

После rollout выполните:

```bash
kubectl -n vethelp-alpha get deployment vethelp-backend
kubectl -n vethelp-alpha get pods -l app.kubernetes.io/name=vethelp-backend
kubectl -n vethelp-alpha logs deployment/vethelp-backend --tail=100
```

Контрольные признаки:

- оба replica находятся в `Running`/`Ready`;
- `/v1/health` возвращает `status: ok`;
- JSON-логи содержат `correlationId` для запросов и worker events;
- ExternalSecret имеет состояние `Ready`;
- отсутствуют ошибки миграций, Secret access, PostgreSQL connectivity и LiveKit/acquiring configuration.

Рекомендуется запустить platform smoke в изолированной Alpha БД из CI runner после deployment:

```bash
cd backend
npm test -- test/platform-smoke.e2e-spec.ts
```

## 8. Rollback и восстановление

### Rollback приложения

```bash
kubectl -n vethelp-alpha rollout undo deployment/vethelp-backend
kubectl -n vethelp-alpha rollout status deployment/vethelp-backend --timeout=180s
```

Rollback приложения не откатывает database migrations автоматически. Миграции являются версионированными контрактами данных; решение об их откате принимает incident commander после оценки совместимости и данных, записанных новой версией.

### Восстановление данных

При порче данных или необходимости восстановления используйте:

```text
backend/docs/runbooks/backup-runbook.md
```

Runbook требует восстановление сначала в изолированную БД, проверку checksum dump, наличие `public.schema_migrations` и `public.schema_migration_checksums`, затем `npm run migrate:verify`. Переключение write traffic выполняется только после одобрения incident commander.

## 9. Release decision checklist

Release manager может открыть контролируемый Alpha rollout, когда подтверждены все пункты:

- [ ] PR #18 одобрен и merge strategy согласована.
- [ ] CI Backend B-01 green для deploy commit.
- [ ] Database backup и restore rehearsal зафиксированы.
- [ ] Secret Manager / ExternalSecret готовы и не содержат placeholder values.
- [ ] Image digest зафиксирован для rollout.
- [ ] Ingress, NetworkPolicy, WAF и CIDR restriction webhook reviewed Operations/Security.
- [ ] Health probes, logs и alert channel проверены в Alpha namespace.
- [ ] Ответственные за incident/rollback назначены.

После выполнения checklist релиз готов к контролируемому закрытому Alpha-тестированию.
